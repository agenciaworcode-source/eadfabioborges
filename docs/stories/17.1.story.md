## Story

**As a** Fábio Borges (administrador da plataforma),
**I want** que o banco de dados suporte o armazenamento de cupons de desconto com todas as suas configurações (tipo, valor, validade, limite de usos e escopo de aplicação),
**so that** as stories 17.2, 17.3 e 17.4 tenham onde persistir e consultar os cupons — sem riscos de perda de dados ou inconsistência.

## Business Value

Cupons de desconto são uma ferramenta comercial essencial para conversão: lançamentos, black friday, parceiros, recuperação de carrinho. Sem uma tabela dedicada com validações no banco (constraints, RLS, unicidade de código), cupons inválidos, expirados ou esgotados poderiam ser aplicados por race conditions ou bugs na camada de aplicação. A foundation segura no banco garante que a lógica de negócio seja enforced independentemente de qualquer falha de código.

## Scope

### IN

#### Tabela `coupons`

Criar tabela com as seguintes colunas:

```
id              uuid PRIMARY KEY DEFAULT gen_random_uuid()
code            text UNIQUE NOT NULL
discount_type   text NOT NULL CHECK (discount_type IN ('percentage', 'fixed'))
discount_value  numeric(10,2) NOT NULL CHECK (discount_value > 0)
max_uses        integer (NULL = ilimitado)
current_uses    integer NOT NULL DEFAULT 0
valid_from      timestamptz NOT NULL DEFAULT now()
valid_until     timestamptz (NULL = sem expiração)
applies_to      text NOT NULL DEFAULT 'all' CHECK (applies_to IN ('all', 'course', 'plan'))
target_id       text (NULL quando applies_to = 'all')
is_active       boolean NOT NULL DEFAULT true
created_at      timestamptz NOT NULL DEFAULT now()
```

#### RLS

- SELECT público: apenas cupons com `is_active = true` (visitantes e alunos só veem cupons ativos)
- ALL admin: admin pode fazer SELECT sem filtro de `is_active`, INSERT, UPDATE, DELETE

#### Index

- `CREATE INDEX ON coupons(code)` — busca por código de cupom é a operação mais frequente
- `CREATE INDEX ON coupons(is_active, valid_until)` — para validação rápida

### OUT

- Seed de cupons de exemplo (responsabilidade do admin via story 17.2)
- Lógica de validação de cupom (fica para 17.3)
- Decremento/incremento de `current_uses` (fica para 17.3 no webhook)
- CHECK constraint adicional para `discount_value <= 100` quando `discount_type = 'percentage'` (validação fica na camada de API — 17.2 e 17.3)

## Acceptance Criteria

- **AC-1:** Após aplicar a migration, `\d coupons` no psql mostra todas as 11 colunas com tipos e defaults corretos
- **AC-2:** O CHECK constraint em `discount_type` aceita apenas `'percentage'` e `'fixed'`; qualquer outro valor retorna erro de constraint
- **AC-3:** O CHECK constraint em `applies_to` aceita apenas `'all'`, `'course'` e `'plan'`; qualquer outro valor retorna erro
- **AC-4:** `code` possui constraint UNIQUE — tentar inserir dois cupons com o mesmo código retorna erro de unicidade
- **AC-5:** Um SELECT sem autenticação ou com usuário não-admin retorna apenas cupons com `is_active = true`
- **AC-6:** Admin pode fazer SELECT em cupons com `is_active = false` (cupons desativados visíveis apenas para admin)
- **AC-7:** Admin pode INSERT um cupom com todos os campos preenchidos; o cupom aparece em `SELECT * FROM coupons`
- **AC-8:** `supabase db push` não retorna nenhum erro
- **AC-9:** Executar a migration duas vezes (idempotência via `IF NOT EXISTS`) não gera erros

## Technical Notes

### SQL da migration

```sql
-- Migration: Epic 17.1 — Tabela coupons

CREATE TABLE IF NOT EXISTS public.coupons (
  id             uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
  code           text        UNIQUE NOT NULL,
  discount_type  text        NOT NULL CHECK (discount_type IN ('percentage', 'fixed')),
  discount_value numeric(10,2) NOT NULL CHECK (discount_value > 0),
  max_uses       integer,                         -- NULL = ilimitado
  current_uses   integer     NOT NULL DEFAULT 0,
  valid_from     timestamptz NOT NULL DEFAULT now(),
  valid_until    timestamptz,                     -- NULL = sem expiração
  applies_to     text        NOT NULL DEFAULT 'all'
                               CHECK (applies_to IN ('all', 'course', 'plan')),
  target_id      text,                            -- NULL quando applies_to = 'all'
  is_active      boolean     NOT NULL DEFAULT true,
  created_at     timestamptz NOT NULL DEFAULT now()
);

-- RLS
ALTER TABLE public.coupons ENABLE ROW LEVEL SECURITY;

-- SELECT público: apenas cupons ativos (para validação no checkout por qualquer usuário)
CREATE POLICY "coupons_read_active_public"
  ON public.coupons FOR SELECT
  USING (is_active = true OR is_admin());

-- Admin tem controle total
CREATE POLICY "coupons_admin_all"
  ON public.coupons FOR ALL
  USING (is_admin());

-- Indexes para performance
CREATE INDEX IF NOT EXISTS idx_coupons_code
  ON public.coupons(code);

CREATE INDEX IF NOT EXISTS idx_coupons_active_valid
  ON public.coupons(is_active, valid_until);
```
