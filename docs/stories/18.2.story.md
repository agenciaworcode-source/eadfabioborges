## Story

**As a** aluno autenticado na plataforma,
**I want** ver meu plano atual e botões para trocar para outros planos em `/dashboard/plano`, com um modal de confirmação mostrando o valor de proration antes de confirmar,
**so that** eu possa fazer upgrade ou downgrade com total transparência sobre o custo da troca.

## Business Value

A UI de mudança de plano é o ponto de conversão direta: sem ela, o backend (18.1) e o webhook (18.3) não têm valor para o usuário. O modal com proration explícito ("você pagará R$X agora") reduz fricção e abandono, já que o aluno sabe exatamente o que vai pagar antes de confirmar. Transparência de preço é um fator comprovado de conversão em SaaS.

## Scope

### IN

- Seção na página `/dashboard/plano` (existente) exibindo:
  - Plano atual do usuário (nome, preço, ciclo de cobrança)
  - Cards ou botões para cada plano disponível com indicação visual de upgrade/downgrade
  - Botão desabilitado no plano atual ("Seu plano atual")
- Modal de confirmação ao clicar em trocar plano:
  - Novo plano selecionado (nome e preço)
  - Valor de proration ("Você pagará R$ X,XX agora" ou "Você receberá crédito de R$ X,XX")
  - Nova data de cobrança
  - Botões "Confirmar troca" e "Cancelar"
- Ao confirmar: chamar `POST /api/plan/change`, exibir feedback de sucesso (toast) e atualizar UI com novo plano sem reload manual
- Estado de loading durante a chamada API
- Tratamento de erro: exibir mensagem de erro no modal se a API retornar erro

### OUT

- Lógica de checkout/pagamento (já tratada pelo Epic 16)
- Tela de gerenciamento de billing completo (portal do cliente Stripe — já existente)
- Suporte a billing semestral (Epic 19)
- Criação da página `/dashboard/plano` do zero (assumida existente do Epic 16)

## Acceptance Criteria

- **AC-1:** Em `/dashboard/plano`, o usuário vê seu plano atual com indicação "Seu plano atual" (botão desabilitado)
- **AC-2:** Os demais planos são exibidos com botão "Fazer upgrade" (plano superior) ou "Fazer downgrade" (plano inferior) — diferenciação visual entre upgrade e downgrade
- **AC-3:** Ao clicar em um plano, abre modal com: nome do novo plano, valor de proration formatado em R$ (positivo = a pagar; negativo = crédito), nova data de cobrança no formato DD/MM/YYYY
- **AC-4:** Durante a chamada à API, o botão "Confirmar troca" exibe estado de loading (spinner/disabled)
- **AC-5:** Após confirmação com sucesso, o modal fecha, um toast de sucesso é exibido ("Plano alterado com sucesso!") e o card do plano atual é atualizado com o novo plano — sem necessidade de refresh manual da página
- **AC-6:** Se a API retornar erro, o modal permanece aberto e exibe a mensagem de erro retornada pela API
- **AC-7:** O modal pode ser fechado sem confirmar (botão "Cancelar" ou clique fora do modal)

### Formatação de proration

```typescript
// prorationAmount em centavos (vindo do Stripe)
const formatProration = (amount: number) => {
  if (amount > 0) return `Você pagará R$ ${(amount / 100).toFixed(2)} agora`
  if (amount < 0) return `Você receberá crédito de R$ ${(Math.abs(amount) / 100).toFixed(2)}`
  return 'Sem cobrança adicional'
}
```

### Busca da proration antes de abrir o modal

O endpoint `POST /api/plan/change` retorna o `prorationAmount` ao executar a troca. Para exibir a proration ANTES da confirmação, o dev deve verificar se existe um endpoint de preview (GET) ou se a proration será buscada ao abrir o modal via POST simulado.

[AUTO-DECISION] Como a Story 18.1 não define endpoint de preview separado, o modal deve exibir a proration já calculada ao confirmar — ou seja, o modal abre com um "Calculando..." e faz a chamada imediatamente ao abrir para obter a proration. Alternativa: buscar proration no clique do plano, antes de abrir o modal.

A implementação mais simples: chamar `POST /api/plan/change` apenas na confirmação final, e exibir no modal apenas as informações estáticas do plano (preço novo vs atual). O `prorationAmount` exato aparece apenas na confirmação. Isso é aceitável e evita lógica extra.
