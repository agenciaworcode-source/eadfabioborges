## Story

**As a** Fábio Borges (administrador da plataforma),
**I want** que as tabelas `courses` e `enrollments` suportem configurações explícitas de tipo de acesso, certificado e prazo de acesso,
**so that** as stories 15.2, 15.3 e 15.4 tenham onde persistir os dados de configuração avançada — sem quebrar cursos e matrículas já existentes.

## Business Value

Atualmente o tipo de acesso ao curso é inferido por `price` e `is_vip` sem nenhum controle explícito. Não há como desabilitar certificado em um curso específico nem limitar o tempo de acesso do aluno. Essa migration é o pré-requisito zero do Epic 15: sem ela, nenhuma das demais stories pode ser implementada. A abordagem backward-compatible com defaults seguros garante que todos os cursos publicados e matrículas ativas continuam funcionando sem necessidade de UPDATE manual.

## Scope

### IN

#### Tabela `courses` — 3 novas colunas

- `access_type text NOT NULL DEFAULT 'paid' CHECK (access_type IN ('free', 'paid', 'plan', 'manual'))` — tipo de acesso explícito
- `certificate_enabled boolean NOT NULL DEFAULT true` — toggle de emissão de certificado
- `access_days integer` (NULL = vitalício) — dias de acesso após matrícula; NULL significa sem expiração

#### Tabela `enrollments` — 1 nova coluna

- `expires_at timestamptz` (NULL = sem expiração) — data/hora de expiração calculada na matrícula

#### Compatibilidade

- Todas as colunas têm DEFAULT seguros: cursos existentes ficam como `access_type = 'paid'`, `certificate_enabled = true`, `access_days = null`
- Matrículas existentes ficam com `expires_at = null` (acesso vitalício)
- Nenhum dado existente é modificado ou removido
- RLS: nenhuma mudança necessária — novas colunas herdam políticas existentes

### OUT

- Nenhuma alteração no código da aplicação (fica para 15.2, 15.3, 15.4)
- Adição do valor `'expired'` ao enum `enrollment_status` (fica para 15.3, junto com a lógica que o usa)
- Seed de dados para diferentes `access_type`
- CHECK constraints adicionais (validação fica na camada de API)
- RLS policies separadas por `access_type`

- **AC-1:** Após aplicar a migration, `\d courses` no psql mostra as colunas `access_type`, `certificate_enabled` e `access_days` com os tipos e defaults corretos
- **AC-2:** O CHECK constraint em `access_type` aceita exatamente os valores `'free'`, `'paid'`, `'plan'` e `'manual'` — qualquer outro valor retorna erro

## Technical Notes

### SQL da migration

```sql
-- Migration: Epic 15 — campos de configuração de curso e expiração de matrícula

-- Adicionar colunas de configuração à tabela courses
ALTER TABLE public.courses
  ADD COLUMN IF NOT EXISTS access_type        text NOT NULL DEFAULT 'paid'
    CHECK (access_type IN ('free', 'paid', 'plan', 'manual')),
  ADD COLUMN IF NOT EXISTS certificate_enabled boolean NOT NULL DEFAULT true,
  ADD COLUMN IF NOT EXISTS access_days        integer;

-- Adicionar coluna de expiração à tabela enrollments
ALTER TABLE public.enrollments
  ADD COLUMN IF NOT EXISTS expires_at timestamptz;
```
