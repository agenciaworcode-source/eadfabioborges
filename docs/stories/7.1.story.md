## Story

**As a** visitante (potencial aluno),
**I want** navegar pelas páginas públicas `/cursos`, `/cursos/[slug]` e `/planos`,
**so that** eu possa descobrir os cursos disponíveis, ver detalhes de cada curso e conhecer os planos de mentoria antes de fazer login ou comprar.

## Business Value

As páginas públicas são o topo do funil da plataforma. Sem `/cursos`, `/cursos/[slug]` e `/planos`, um visitante que chega pelo link só vê a home — e a navegação do `PublicNav` (que já tem os links `/cursos` e `/planos`) retorna 404. Isso bloqueia qualquer conversão. Epic 7 fecha o funil público antes de integrar pagamentos (Sprint 3, deferido).

## Scope

### IN

- **`/cursos`** (`src/app/cursos/page.tsx`) — Server Component; busca todos os cursos com `published = true` no Supabase; exibe grid de cards com título, preço, badge VIP e link para `/cursos/[slug]`
- **`/cursos/[slug]`** (`src/app/cursos/[slug]/page.tsx`) — Server Component; busca curso pelo `slug` + contagem de módulos e aulas; exibe título, descrição, preço, badge VIP, lista de módulos (títulos); CTA via `<CheckoutButton>` (deferido — Sprint 3); retorna `notFound()` se slug inexistente

- Metadados (`export const metadata`) com `title` e `description` para SEO básico

- Paginação do catálogo

### Padrão de Server Component com Supabase

Seguir exatamente o padrão de `src/app/certificado/[uuid]/page.tsx`:

```typescript
import { notFound } from 'next/navigation'
import { createClient } from '@/lib/supabase/server'

export default async function Page() {
  const supabase = createClient()
  const { data, error } = await supabase
    .from('courses')
    .select('id, slug, title, description, price, is_vip, thumbnail_url')
    .eq('published', true)
    .order('created_at', { ascending: false })

  if (error) throw error
  // render...
}
```

### Query para /cursos/[slug] — módulos e aulas

```typescript
const { data: course } = await supabase

    modules(id, title, order, lessons(id))

  .eq('slug', params.slug)
  .eq('published', true)

if (!course) notFound()
```

### Formatação de preço

```typescript
function formatPrice(price: number | null, isVip: boolean): string {
  if (isVip) return 'Incluso no plano'
  if (!price) return 'Gratuito'
  return new Intl.NumberFormat('pt-BR', { style: 'currency', currency: 'BRL' }).format(price)
}
```

### Arquitetura Checkout-Ready (plug-in play)

O checkout está deferido (Sprint 3), mas as páginas públicas devem ser construídas de forma que **qualquer provider de checkout** (Stripe, MercadoPago, Hotmart, etc.) possa ser plugado sem alterar as páginas.

**Padrão:** componente `<CheckoutButton>` em `src/components/checkout/CheckoutButton.tsx`:

```typescript
// src/components/checkout/CheckoutButton.tsx
'use client'

interface CheckoutButtonProps {
  label: string
  courseSlug?: string   // para cursos avulsos
  planId?: string       // para planos ('prata' | 'ouro' | 'diamante' | 'macroempresa')
  className?: string
}

export function CheckoutButton({ label, courseSlug, planId, className }: CheckoutButtonProps) {
  // FASE 1 (atual): redireciona para login com redirect_to
  // FASE 2 (Sprint 3): substituir o body desta função para chamar /api/checkout/curso ou /api/checkout/plano
  const href = courseSlug
    ? `/auth/login?redirect=/cursos/${courseSlug}`
    : planId
    ? `/auth/login?redirect=/planos`
    : '/auth/login'

  return (
    <a href={href} className={className}>
      {label}
    </a>
  )
}
```

**Por que esse padrão:**

- As páginas `/cursos/[slug]` e `/planos` apenas chamam `<CheckoutButton courseSlug={course.slug} label="Quero me matricular" />` — nunca sabem o provider
- Na Sprint 3, o time só edita `CheckoutButton.tsx` (troca o `href` por um `fetch` ao `/api/checkout/curso`) — zero alteração nas páginas
- Extensível: adicionar `provider?: 'stripe' | 'mercadopago'` prop sem breaking change
