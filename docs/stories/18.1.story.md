## Story

**As a** aluno autenticado na plataforma,
**I want** poder trocar meu plano de assinatura via API com cálculo automático de proration,
**so that** a diferença de valor seja cobrada corretamente no momento da troca, sem precisar cancelar e criar uma nova assinatura.

## Business Value

A troca de plano com proration é um recurso crítico de retenção: sem ela, o aluno precisaria cancelar e recriar a assinatura, gerando atrito e risco de perda. O uso de `proration_behavior: 'create_prorations'` do Stripe garante cobrança justa (o aluno paga apenas a diferença pro-rata do período restante), aumentando conversão de upgrade e reduzindo churn em downgrades.

## Scope

### IN

- Endpoint `POST /api/plan/change` recebendo `{ newPlanId: string, billingPeriod: 'monthly' | 'annual' }` via body
- Autenticação obrigatória via Supabase session (middleware existente)
- Buscar subscription ativa do usuário via `stripe.subscriptions.retrieve` usando o `stripe_subscription_id` armazenado no banco
- Atualizar subscription via `stripe.subscriptions.update` com:
  - Novo `price_id` correspondente a `newPlanId` + `billingPeriod`
  - `proration_behavior: 'create_prorations'`
- Retornar `{ success: true, prorationAmount: number, nextBillingDate: string }`
- Tratamento de erros: subscription não encontrada → 400; usuário trocando para o mesmo plano → 400; erro Stripe → 500

### OUT

- Atualização do banco de dados após troca (responsabilidade do webhook 18.3)
- UI de confirmação (Story 18.2)
- Lógica de cálculo de proration no frontend (Stripe retorna o valor)
- Suporte a billing semestral (Epic 19)

## Dependencies

- **Epic 16 deve estar completo** — campo `stripe_subscription_id` em `users` ou `subscriptions` é pré-requisito
- Variáveis de ambiente: `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET` (já presentes do Epic 16)
- Tabela `plans` com campo `stripe_price_id_monthly` e `stripe_price_id_annual`

## Acceptance Criteria

- **AC-1:** `POST /api/plan/change` com `{ newPlanId: 'ouro', billingPeriod: 'monthly' }` retorna `200` com `{ success: true, prorationAmount: N, nextBillingDate: "YYYY-MM-DD" }` quando o usuário possui subscription ativa e o plano destino é diferente do atual
- **AC-2:** Upgrade de plano mensal → anual funciona: a subscription no Stripe é atualizada com o novo price_id e `proration_behavior: 'create_prorations'`
- **AC-3:** Downgrade de plano 'diamante' → 'prata' funciona: subscription atualizada com proration
- **AC-4:** Request sem subscription ativa (usuário nunca assinou ou cancelou) retorna `400` com `{ error: 'No active subscription found' }`
- **AC-5:** Request tentando trocar para o mesmo plano atual retorna `400` com `{ error: 'Already on this plan' }`
- **AC-6:** Request sem autenticação retorna `401`
- **AC-7:** `prorationAmount` no response reflete o valor calculado pelo Stripe (pode ser positivo para upgrades ou negativo para downgrades/créditos)

### Lógica do endpoint

```typescript
// POST /api/plan/change
import { createRouteHandlerClient } from '@supabase/auth-helpers-nextjs'
import { cookies } from 'next/headers'
import Stripe from 'stripe'

const stripe = new Stripe(process.env.STRIPE_SECRET_KEY!)

export async function POST(request: Request) {
  const supabase = createRouteHandlerClient({ cookies })
  const {
    data: { session },
  } = await supabase.auth.getSession()

  if (!session) return Response.json({ error: 'Unauthorized' }, { status: 401 })

  const { newPlanId, billingPeriod } = await request.json()

  // Buscar subscription ativa do usuário
  const { data: subscription } = await supabase
    .from('subscriptions')
    .select('stripe_subscription_id, plan_id, billing_period')
    .eq('user_id', session.user.id)
    .eq('status', 'active')
    .single()

  if (!subscription)
    return Response.json({ error: 'No active subscription found' }, { status: 400 })
  if (subscription.plan_id === newPlanId && subscription.billing_period === billingPeriod) {
    return Response.json({ error: 'Already on this plan' }, { status: 400 })
  }

  // Buscar price_id do novo plano
  const { data: plan } = await supabase
    .from('plans')
    .select('stripe_price_id_monthly, stripe_price_id_annual')
    .eq('id', newPlanId)
    .single()

  const newPriceId =
    billingPeriod === 'annual' ? plan.stripe_price_id_annual : plan.stripe_price_id_monthly

  // Buscar subscription atual no Stripe
  const stripeSub = await stripe.subscriptions.retrieve(subscription.stripe_subscription_id)

  // Atualizar subscription com proration
  const updatedSub = await stripe.subscriptions.update(subscription.stripe_subscription_id, {
    items: [{ id: stripeSub.items.data[0].id, price: newPriceId }],
    proration_behavior: 'create_prorations',
  })

  // Calcular prorationAmount a partir dos upcoming invoice items
  const upcomingInvoice = await stripe.invoices.retrieveUpcoming({
    customer: stripeSub.customer as string,
    subscription: subscription.stripe_subscription_id,
  })

  const prorationAmount = upcomingInvoice.amount_due
  const nextBillingDate = new Date(updatedSub.current_period_end * 1000).toISOString().slice(0, 10)

  return Response.json({ success: true, prorationAmount, nextBillingDate })
}
```

### Campos necessários no banco

- `subscriptions.stripe_subscription_id` — ID da subscription Stripe do usuário
- `subscriptions.plan_id` — plano atual
- `subscriptions.billing_period` — 'monthly' | 'annual'
- `plans.stripe_price_id_monthly` e `plans.stripe_price_id_annual` — price IDs no Stripe

### Padrão de error handling

Seguir padrão dos outros endpoints do Epic 16: try/catch com `console.error` e Response.json com status HTTP correto.
