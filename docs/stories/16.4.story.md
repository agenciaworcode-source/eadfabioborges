## Story

**As a** visitante ou aluno da plataforma,
**I want** ver quais cursos estão inclusos em cada plano na página de planos, e ver um badge "Incluso no Plano X" na página do curso quando ele fizer parte de um plano,
**so that** eu consiga tomar uma decisão de compra informada sem precisar navegar entre múltiplas páginas para entender o que cada plano oferece.

## Business Value

Toda a infraestrutura do Epic 16 (tabela, admin, função) é invisível para o usuário final sem essa story. Um visitante que acessa a página de planos precisa ver claramente o que está incluído em cada pacote para decidir qual assinar. Um aluno que acessa a página de um curso precisa saber se ele já está coberto pelo seu plano — evitando uma compra avulsa desnecessária. Essa story fecha o ciclo de comunicação do Epic 16.

## Scope

### IN

#### Página `/planos` — lista de cursos por plano

- Localizar a página pública de planos (ex: `src/app/(public)/planos/page.tsx` ou equivalente)
- Para cada plano listado: exibir os cursos inclusos via query `plan_courses JOIN courses`
- Exibir: título do curso e (opcionalmente) thumbnail pequena
- Se um plano não tiver cursos em `plan_courses`: exibir "Nenhum curso disponível"
- Ordenar cursos por título dentro de cada plano

#### Página `/cursos/[slug]` — badge "Incluso no Plano X"

- Localizar a página pública de detalhe do curso
- Quando `access_type = 'plan'` (campo adicionado no Epic 15): buscar em `plan_courses` quais planos incluem esse curso
- Exibir badge/tag "Incluso no Plano Prata", "Incluso nos Planos Prata e Ouro" etc. na área de CTA do curso
- Se o curso não estiver em nenhum plano (ou `access_type != 'plan'`): não exibir o badge

### OUT

- Compra ou assinatura de plano direto dessas páginas (apenas exibição informativa)
- Filtro de cursos por plano na listagem geral de cursos
- Comparativo lado a lado de planos (feature separada)
- Indicação de "você já tem acesso" via autenticação (apenas exibição pública, sem lógica de usuário logado)

## Acceptance Criteria

- **AC-1:** Na página `/planos`, cada plano exibe a lista de cursos inclusos carregada dinamicamente de `plan_courses JOIN courses WHERE published = true`
- **AC-2:** A lista de cursos em cada plano é ordenada por título do curso (A-Z)
- **AC-3:** Um plano sem cursos cadastrados em `plan_courses` exibe a mensagem "Nenhum curso disponível"
- **AC-4:** Na página `/cursos/[slug]` de um curso com `access_type = 'plan'` que está em `plan_courses`, aparece um badge visível com texto "Incluso no Plano [NomeDoPLano]" (ou lista de planos separados por vírgula/e se mais de um)
- **AC-5:** Na página `/cursos/[slug]` de um curso com `access_type != 'plan'` (ex: `'paid'`, `'free'`), nenhum badge de plano é exibido
- **AC-6:** As queries para `plan_courses` usam o cliente Supabase público (sem autenticação necessária) — a RLS de SELECT público da tabela permite isso
- **AC-7:** `npm run build`, `npm run lint` e `npm run typecheck` passam sem erros
- **AC-8:** Não há erros no console do browser ao carregar as páginas de planos e de detalhe do curso

## Technical Notes

### Query para página de planos

```typescript
// Buscar cursos de um plano
const { data: planCourses } = await supabase
  .from('plan_courses')
  .select(
    `
    course_id,
    courses (
      id,
      title,
      thumbnail_url,
      slug
    )
  `
  )
  .eq('plan_id', plan.id)
  .order('courses(title)')
```

### Query para badge na página do curso

```typescript
// Buscar planos que incluem este curso
const { data: plansWithCourse } = await supabase
  .from('plan_courses')
  .select(
    `
    plan_id,
    plans (
      id,
      name
    )
  `
  )
  .eq('course_id', course.id)

// Exibir badge apenas se access_type === 'plan' E plansWithCourse.length > 0
```

### Tratamento de TypeScript para relações Supabase

Ao usar queries com JOINs via `.select()`, o tipo retornado pode ser `courses: { id, title, ... } | null`. Usar cast explícito conforme padrão do projeto para evitar erros de tipo `never`.
