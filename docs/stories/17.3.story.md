## Story

**As a** aluno da plataforma,
**I want** que o desconto do meu cupom seja aplicado corretamente ao preço durante o checkout,
**so that** eu pague o valor com desconto e não o preço cheio — e para que cupons inválidos, expirados ou de outro produto sejam rejeitados com uma mensagem clara antes de eu tentar pagar.

## Business Value

Sem a validação e aplicação do cupom no backend, qualquer cupom que chegue ao frontend pode ser aceito ou rejeitado por inconsistências de estado. Pior: sem o incremento atômico de `current_uses` no webhook de pagamento, cupons com limite de uso poderiam ser usados mais vezes do que o permitido. Essa story garante que a lógica financeira do desconto seja confiável e auditável — o que é essencial para não gerar prejuízo em promoções de alto volume.

## Scope

### IN

#### API `POST /api/checkout/validate-coupon`

- Endpoint público (autenticação opcional — necessário apenas para checkout)
- Body: `{ code: string, itemType: 'course' | 'plan', itemId: string }`
- Validações em sequência:
  1. Cupom existe e `is_active = true`
  2. `valid_from <= now()` e (`valid_until IS NULL` OR `valid_until > now()`)
  3. `max_uses IS NULL` OR `current_uses < max_uses`
  4. `applies_to = 'all'` OU (`applies_to = itemType` E `target_id = itemId`)
- Resposta de sucesso (200): `{ valid: true, discountType: string, discountValue: number, discountedPrice: number }`
- Resposta de erro (422): `{ valid: false, error: string }` com mensagem específica por tipo de falha

#### Aplicação do desconto nas rotas de checkout existentes

- `src/app/api/checkout/curso/route.ts`: aceitar `couponCode?: string` no body
  - Se `couponCode` presente: chamar validate-coupon internamente e aplicar ao `unit_amount` antes de criar a session Stripe/MP
  - Se inválido no momento do checkout: rejeitar com erro (não criar session)
- `src/app/api/checkout/plano/route.ts`: mesma lógica para planos

#### Incremento de `current_uses` após pagamento confirmado

- Localizar os webhooks de Stripe e/ou MercadoPago em `src/app/api/`
- Após confirmação de pagamento bem-sucedido: buscar o `coupon_code` associado à sessão e executar:
  `UPDATE coupons SET current_uses = current_uses + 1 WHERE code = $1`
- Usar UPDATE com WHERE como operação atômica (sem necessidade de SELECT + UPDATE separados)

### OUT

- Stripe Promotions API (usar desconto direto no `unit_amount` para manter simplicidade)
- Fila de processamento assíncrono do incremento (operação síncrona no webhook é suficiente)
- Reembolso ou estorno ao invalidar cupom pós-compra
- Testes automatizados (ACs validados manualmente)

## Acceptance Criteria

- **AC-1:** `POST /api/checkout/validate-coupon` com código de cupom ativo, dentro da validade, não esgotado e aplicável ao item retorna 200 com `{ valid: true, discountType, discountValue, discountedPrice }`
- **AC-2:** Cupom com `valid_until < now()` retorna 422 com `{ valid: false, error: 'Cupom expirado' }`
- **AC-3:** Cupom com `current_uses >= max_uses` retorna 422 com `{ valid: false, error: 'Cupom esgotado' }`
- **AC-4:** Cupom com `applies_to = 'course'` e `target_id = course_A` retorna 422 com erro "Cupom não aplicável a este produto" quando tentado no `course_B`
- **AC-5:** Cupom com `applies_to = 'all'` é aceito para qualquer curso ou plano
- **AC-6:** Código de cupom inexistente retorna 422 com `{ valid: false, error: 'Cupom inválido' }`
- **AC-7:** `POST /api/checkout/curso` com `couponCode` válido cria uma sessão Stripe/MP com `unit_amount` reduzido pelo desconto calculado corretamente (desconto % ou valor fixo)
- **AC-8:** `POST /api/checkout/curso` com `couponCode` inválido (qualquer das condições de AC-2 a AC-6) retorna 400 e não cria sessão de pagamento
- **AC-9:** Após webhook de pagamento confirmado disparado para uma compra com cupom, `SELECT current_uses FROM coupons WHERE code = 'CUPOM'` mostra incremento de +1 em relação ao valor anterior
- **AC-10:** `npm run build`, `npm run lint` e `npm run typecheck` passam sem erros

## Technical Notes

### Endpoint validate-coupon

```typescript
// src/app/api/checkout/validate-coupon/route.ts

export async function POST(request: Request) {
  const { code, itemType, itemId }: {
    code: string
    itemType: 'course' | 'plan'
    itemId: string
  } = await request.json()

  const supabase = createClient()
  const { data: coupon } = await supabase
    .from('coupons')
    .select('*')
    .eq('code', code.toUpperCase())
    .eq('is_active', true)
    .single()

  if (!coupon) {
    return NextResponse.json({ valid: false, error: 'Cupom inválido' }, { status: 422 })
  }

  const now = new Date()

  if (new Date(coupon.valid_from) > now) {
    return NextResponse.json({ valid: false, error: 'Cupom ainda não está ativo' }, { status: 422 })
  }
  if (coupon.valid_until && new Date(coupon.valid_until) <= now) {
    return NextResponse.json({ valid: false, error: 'Cupom expirado' }, { status: 422 })
  }
  if (coupon.max_uses !== null && coupon.current_uses >= coupon.max_uses) {
    return NextResponse.json({ valid: false, error: 'Cupom esgotado' }, { status: 422 })
  }
  if (coupon.applies_to !== 'all') {
    if (coupon.applies_to !== itemType || coupon.target_id !== itemId) {
      return NextResponse.json({ valid: false, error: 'Cupom não aplicável a este produto' }, { status: 422 })
    }
  }

  // Calcular preço com desconto
  // itemPrice deve ser buscado do banco conforme itemType e itemId
  // ...

  return NextResponse.json({ valid: true, discountType: coupon.discount_type, discountValue: coupon.discount_value, discountedPrice: ... })
}
```

### Cálculo do desconto

```typescript
function applyDiscount(originalPrice: number, type: string, value: number): number {
  if (type === 'percentage') {
    return Math.round(originalPrice * (1 - value / 100))
  }
  // fixed: desconto em centavos (o sistema usa centavos internamente)
  return Math.max(0, originalPrice - Math.round(value * 100))
}
```

### Propagação do couponCode para o webhook

Para rastrear qual cupom foi usado em cada compra, o `couponCode` deve ser passado via `metadata` na criação da session Stripe:

```typescript
// Na criação da session Stripe (simplificado)

  // ...
  metadata: {
    coupon_code: couponCode ?? '',
    // outros metadados existentes
  },
})
```

No webhook, recuperar via `session.metadata.coupon_code`.

### Incremento atômico no webhook

```typescript
// Após confirmar pagamento bem-sucedido
const couponCode = session.metadata?.coupon_code
if (couponCode) {
  await supabase
    .from('coupons')
    .update({ current_uses: supabase.rpc('increment_current_uses', { p_code: couponCode }) })
  // Alternativa sem RPC — usando raw SQL via rpc:
  // supabase.rpc('increment_coupon_uses', { p_code: couponCode })
}
```

**Alternativa mais simples** (sem RPC separada):

```sql
-- Executar no webhook via supabase.rpc ou query direta
UPDATE public.coupons
SET current_uses = current_uses + 1
WHERE code = $1 AND is_active = true;
```

## CodeRabbit Integration

> **Story Type:** Feature (Backend/API)
> **Primary Type:** Feature
> **Complexity:** Medium

**Self-Healing Configuration:**

- Primary Agent: @dev (YOLO mode)
- Max Iterations: 2
- Severity Filter: CRITICAL, HIGH

**Focus Areas:**

- Atomicidade do incremento: `UPDATE ... SET current_uses = current_uses + 1` é atômico — nunca usar SELECT + UPDATE em duas queries separadas
- Validação de sequência: expiração deve ser checada ANTES do limite de usos — retornar o erro mais específico
- Não criar session de pagamento com preço negativo: `Math.max(0, ...)` no cálculo de desconto fixo
- Normalizar código para UPPERCASE na busca: `.eq('code', code.toUpperCase())`
- Webhook idempotência: se o webhook disparar duas vezes para o mesmo pagamento, `current_uses` não deve incrementar duas vezes — considerar armazenar o `session_id` do pagamento para deduplicação
