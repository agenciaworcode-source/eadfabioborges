## Story

**As a** aluno autenticado na plataforma EAD Mentoria Fábio Borges,
**I want** ver claramente quais aulas estão disponíveis para o meu plano e ser direcionado para o upgrade quando tentar acessar conteúdo bloqueado,
**so that** eu entenda o valor de cada plano sem frustração, e o Fábio converta mais alunos para planos pagos através de CTAs contextuais.

## Business Value

Sem o AccessGate, dois problemas críticos ocorrem: (1) o banco bloqueia o acesso via RLS mas o usuário não recebe feedback visual — apenas um erro vazio; (2) não há nenhum incentivo de upgrade contextual no momento de maior intenção de compra (quando o aluno tenta acessar uma aula bloqueada). Esta story fecha o Sprint 2 e é o **motor de conversão da plataforma** — cada CTA de upgrade no AccessGate é uma oportunidade de venda direta.

**IN (o que esta story entrega):**

## Acceptance Criteria

1. Hook `useEnrollment(courseId: string)` criado em `src/hooks/use-enrollment.ts`:
   - Retorna `{ hasAccess: boolean, isLoading: boolean, error: Error | null }`
   - `hasAccess = true` se EXISTS enrollment com `status IN ('active', 'completed')` para o curso
   - `hasAccess = true` se EXISTS subscription com `status = 'active'` E `period_end > now()`
   - Usa `createBrowserClient()` de `src/lib/supabase/client.ts`
   - Executa as duas queries em paralelo com `Promise.all`

2. Componente `<AccessGate>` criado em `src/components/shared/AccessGate.tsx`:
   - Props: `courseId: string`, `children: React.ReactNode`, `courseName?: string`, `planRequired?: 'prata' | 'ouro' | 'diamante'`
   - Quando `isLoading`: renderiza `<LoadingSkeleton>` (ou div com `animate-pulse`)
   - Quando `hasAccess: true`: renderiza `children` diretamente sem qualquer wrapper
   - Quando `hasAccess: false`: renderiza overlay sobre children com:
     - Ícone de cadeado (Lucide `Lock`)
     - Texto: "Conteúdo disponível no plano {planRequired ?? 'Prata'} ou superior"
     - Botão primário "Ver planos" → link para `/planos`
     - Botão secundário "Comprar curso avulso" → link para `/cursos/{courseSlug}` (se `courseId` disponível)

3. Página `/dashboard/curso/[id]/page.tsx` criada em `src/app/(dashboard)/dashboard/curso/[id]/page.tsx`:
   - Server Component que busca dados do curso (título, módulos, aulas)
   - Envolve a lista de aulas em `<AccessGate courseId={id}>`
   - Exibe módulos e aulas sem o player (player será implementado na Sprint 4)
   - Aulas bloqueadas pelo AccessGate mostram cadeado em vez de botão play

4. Cenários de acesso testados (testes unitários em `tests/unit/hooks/use-enrollment.test.ts`):
   - ✅ Usuário com enrollment `status: 'active'` → `hasAccess: true`
   - ✅ Usuário com enrollment `status: 'completed'` → `hasAccess: true`
   - ✅ Usuário com subscription `status: 'active'` e `period_end` futuro → `hasAccess: true`
   - ✅ Usuário sem enrollment nem subscription → `hasAccess: false`
   - ✅ Usuário com enrollment `status: 'refunded'` → `hasAccess: false`
   - ✅ Usuário com subscription `status: 'canceled'` → `hasAccess: false`

5. `<AccessGate>` é um Client Component (`'use client'`) pois usa hook com estado

6. Nenhum dado de curso é vazado para usuários sem acesso — o overlay cobre o children mas não os expõe no DOM com dados sensíveis (a lista de aulas pode ser exibida como títulos públicos; o conteúdo do vídeo só existe na Sprint 4)

### Regra de Acesso no Banco (RLS — já implementada)

[Source: docs/architecture/9-schema-do-banco-de-dados.md — 9.2 RLS Policies Críticas]

A tabela `lessons` já tem RLS que rejeita acesso sem enrollment ou subscription ativa. O `AccessGate` é a **camada de UX** — fornece feedback visual antes da query falhar silenciosamente. A query do hook deve espelhar exatamente a mesma lógica da RLS:

```sql
-- RLS já implementada (referência para o hook)
-- enrollment ativo OU subscription ativa com period_end futuro
EXISTS (SELECT 1 FROM enrollments WHERE user_id = auth.uid() AND course_id = ... AND status IN ('active', 'completed'))
OR
EXISTS (SELECT 1 FROM subscriptions WHERE user_id = auth.uid() AND status = 'active' AND period_end > now())
```

### Padrão de Hook com Supabase (referência da Story 2.1)

[Source: docs/stories/2.1.story.md — Task 3]

```typescript
// src/hooks/use-enrollment.ts
'use client'
import { useState, useEffect } from 'react'
import { createClient } from '@/lib/supabase/client'

export function useEnrollment(courseId: string) {
  const [hasAccess, setHasAccess] = useState(false)
  const [isLoading, setIsLoading] = useState(true)
  const [error, setError] = useState<Error | null>(null)

  useEffect(() => {
    async function checkAccess() {
      try {
        const supabase = createClient()
        const {
          data: { user },
        } = await supabase.auth.getUser()
        if (!user) {
          setHasAccess(false)
          setIsLoading(false)
          return
        }

        const [{ data: enrollments }, { data: subscriptions }] = await Promise.all([
          supabase
            .from('enrollments')
            .select('status')
            .eq('user_id', user.id)
            .eq('course_id', courseId)
            .in('status', ['active', 'completed']),
          supabase
            .from('subscriptions')
            .select('status, period_end')
            .eq('user_id', user.id)
            .eq('status', 'active')
            .gt('period_end', new Date().toISOString()),
        ])

        setHasAccess((enrollments?.length ?? 0) > 0 || (subscriptions?.length ?? 0) > 0)
      } catch (err) {
        setError(err instanceof Error ? err : new Error('Erro ao verificar acesso'))
        setHasAccess(false)
      } finally {
        setIsLoading(false)
      }
    }
    checkAccess()
  }, [courseId])

  return { hasAccess, isLoading, error }
}
```
