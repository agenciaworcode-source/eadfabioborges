## Story

**As a** Fábio Borges (administrador da plataforma),
**I want** uma seção "Configurações de Acesso" no formulário de criação/edição de curso com controles para tipo de acesso, certificado e prazo,
**so that** eu possa definir explicitamente como cada curso é acessado, se emite certificado e por quantos dias o aluno tem acesso após se matricular.

## Business Value

Sem essa interface, os campos introduzidos pela Story 15.1 ficam inacessíveis via painel admin — só poderiam ser configurados via SQL direto. Ao centralizar os controles no editor de curso existente (`AdminCursosEditor`), o Fábio ganha autonomia total sobre a política de acesso de cada curso sem precisar de intervenção técnica. A lógica de mostrar/ocultar o campo de preço conforme o `access_type` elimina inconsistências (ex: curso gratuito com preço preenchido).

## Scope

### IN

#### Seção "Configurações de Acesso" no `AdminCursosEditor`

- **Select `access_type`** com 4 opções:
  - `free` → "Gratuito"
  - `paid` → "Pago"
  - `plan` → "Por Plano (assinatura)"
  - `manual` → "Manual (admin atribui)"
- **Toggle `certificate_enabled`**: label "Emitir certificado ao concluir" — checked por padrão
- **Input numérico `access_days`**: label "Dias de acesso após matrícula" — placeholder "0 = vitalício"; aceita inteiros >= 1 ou campo vazio (vitalício); valor 0 ou vazio salvo como NULL

#### Lógica de visibilidade do campo de preço

- Campo `price` visível somente quando `access_type === 'paid'`
- Ao trocar para qualquer outro tipo, o campo de preço some do formulário (mas mantém o valor no state para caso o admin volte para 'paid')

#### Persistência via API existente

- Incluir `access_type`, `certificate_enabled` e `access_days` no PATCH de `src/app/api/admin/courses/[id]/route.ts`
- Incluir os campos no SELECT da página `src/app/(admin)/admin/cursos/page.tsx`
- Atualizar a interface `CourseRow` em `AdminCursosEditor.tsx` para incluir os 3 novos campos

### OUT

- Nenhuma lógica de expiração de matrícula (fica para Story 15.3)
- Nenhuma verificação de certificado no frontend do aluno (fica para Story 15.4)
- Validação de regra de negócio no banco (fica na camada de API)
- Migração de cursos existentes para um `access_type` diferente de 'paid'

## Acceptance Criteria

- **AC-1:** A seção "Configurações de Acesso" aparece no formulário de edição de curso, abaixo das configurações existentes
- **AC-2:** O select de `access_type` exibe exatamente 4 opções (Gratuito / Pago / Por Plano / Manual) e persiste o valor ao salvar — um GET do curso após salvar retorna o `access_type` correto
- **AC-3:** O campo de preço (`price`) fica visível somente quando `access_type === 'paid'`; ao trocar para outro tipo, o campo de preço desaparece da UI
- **AC-4:** O toggle `certificate_enabled` inicia marcado (true) para cursos sem valor salvo; ao desmarcar e salvar, um GET do curso retorna `certificate_enabled = false`
- **AC-5:** O input `access_days` aceita somente inteiros >= 1; valor vazio ou 0 é enviado como `null` para a API (campo salvo como NULL no banco)
- **AC-6:** Ao salvar um curso com `access_days = 30`, o campo persiste com valor 30 e reabre o formulário exibindo "30" no input
- **AC-7:** `npm run build`, `npm run lint` e `npm run typecheck` passam sem erros
