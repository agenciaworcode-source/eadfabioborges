## Story

**As a** Fábio Borges (administrador da plataforma),
**I want** que o painel admin gerencie cursos de forma completa e segura — incluindo exclusão de cursos, marcação VIP, reordenamento de módulos/aulas e validação de inputs em todas as APIs,
**so that** não haja inconsistências de dados nem operações bloqueadas por falta de endpoint.

## Business Value

Com o Epic 6 entregue (Course Builder, Quiz e Assignment), o CMS admin está funcional — mas tem gaps críticos que impedem uso real em produção:

- Admin não consegue deletar um curso criado errado
- Campo `is_vip` (acesso exclusivo por plano) não é editável pelo painel
- APIs de módulo/aula/quiz/assignment aceitam qualquer campo sem validação (mass assignment)
- Não há como reordenar módulos ou aulas após criação

Fechar esses gaps é pré-requisito para o Epic 7 (páginas públicas e player) e para o hardening de produção (Sprint 7 do PRD).

## Scope

### IN

- **DELETE `/api/admin/courses/[id]`**: excluir curso (cascata na DB via FK → módulos, aulas, quizzes, assignments)
- **PATCH `/api/admin/courses/[id]`**: incluir campo `is_vip` no bodySchema Zod existente
- **Validação Zod** nos 4 PATCHs atualmente sem schema:
  - `PATCH /api/admin/modules/[id]` — aceitar somente `title`, `order`, `is_free_preview`
  - `PATCH /api/admin/lessons/[id]` — aceitar somente `title`, `vimeo_id`, `duration_secs`, `order`
  - `PATCH /api/admin/quizzes/[id]` — aceitar somente `title`, `pass_score`, `attempts_allowed`
  - `PATCH /api/admin/assignments/[id]` — aceitar somente `title`, `instructions`, `deadline`
- **Endpoint de reordenamento em batch**:
  - `PATCH /api/admin/courses/[courseId]/modules/reorder` — body: `{ order: [{ id, order }] }`
  - `PATCH /api/admin/modules/[moduleId]/lessons/reorder` — body: `{ order: [{ id, order }] }`
- **Frontend `AdminCursosEditor`**: botão "Excluir Curso" com confirmação modal + suporte ao campo `is_vip` no form de criação/edição

### OUT

- Upload de thumbnail (requer Supabase Storage — escopo separado, Epic 7+)
- Reordenamento por drag-and-drop (UX avançado — out of scope desta story, só o endpoint)
- Soft-delete de curso (exclusão é hard delete com cascata)
- Endpoint `GET /api/admin/courses/[id]` (página de cursos já usa Supabase server-side diretamente — não necessário)

### Padrão requireAdmin já existente

Todos os novos endpoints devem usar a função `requireAdmin()` já padronizada nas rotas do Epic 6:

```typescript
async function requireAdmin() {
  const supabase = createClient()
  const {
    data: { user },
  } = await supabase.auth.getUser()
  if (!user)
    return { error: NextResponse.json({ error: 'Unauthorized' }, { status: 401 }), supabase: null }
  const { data: profile } = await supabase.from('users').select('role').eq('id', user.id).single()
  if ((profile as { role: string } | null)?.role !== 'admin')
    return { error: NextResponse.json({ error: 'Forbidden' }, { status: 403 }), supabase: null }
  return { error: null, supabase }
}
```

[Source: src/app/api/admin/courses/[courseId]/modules/route.ts]

### Zod strict() para prevenir mass assignment

Usar `.strict()` em todos os schemas de PATCH para rejeitar campos extras:

```typescript
const patchModuleSchema = z
  .object({
    title: z.string().min(1).optional(),
    order: z.number().int().min(0).optional(),
    is_free_preview: z.boolean().optional(),
  })
  .strict()
```

[Source: docs/architecture/15-segurana-e-performance.md — "Validação Zod em 100% das API routes"]
[Source: docs/prd/7-sprints.md Sprint 7 — "Validação Zod em 100% das API routes (nunca confiar no cliente)"]

### Padrão TypeScript para updates Supabase

Usar `as never` ao passar updates para o Supabase client (padrão estabelecido no projeto):

```typescript
await supabase!
  .from('modules')
  .update(parsed.data as never)
  .eq('id', params.id)
```

### Endpoint de reordenamento

Body esperado para ambos os endpoints de reorder:

```typescript
const reorderSchema = z.object({
  order: z
    .array(
      z.object({
        id: z.string().uuid(),
        order: z.number().int().min(0),
      })
    )
    .min(1),
})
```

Implementar como batch de updates individuais (sem transaction nativa no Supabase JS SDK — usar `Promise.all`).
