## Story

**As a** aluno autenticado com acesso a uma aula,
**I want** responder um quiz ao final da aula,
**so that** possa avaliar meu aprendizado, receber feedback imediato sobre minha nota, e ter a próxima aula desbloqueada automaticamente ao ser aprovado.

## Business Value

O quiz é o mecanismo de retenção e validação de aprendizado da plataforma — sem ele, o aluno passa pelas aulas sem confirmar o conhecimento adquirido. A lógica de bloqueio (próxima aula só após aprovação no quiz) cria engajamento obrigatório e aumenta a taxa de conclusão real dos cursos. O sistema também fornece ao admin dados sobre quais aulas os alunos têm mais dificuldade (via taxa de aprovação), alimentando relatórios futuros (Sprint 6).

## Scope

**IN (o que esta story entrega):**

- Componente `<Quiz>` com `'use client'` — renderiza questões de múltipla escolha, V/F e dissertativa
- Componente `<QuizResult>` — exibe nota, status passou/reprovado, gabarito (quando configurado) e botão de tentar novamente
- API route `POST /api/quiz/submit` — calcula nota, verifica `attempts_allowed`, salva `quiz_attempt`, retorna `{ score, passed, correctAnswers? }`
- Hook `useQuiz(lessonId)` — busca quiz + questões da aula via TanStack Query, busca última tentativa do user
- Integração no `CoursePlayerLayout` — exibir quiz abaixo do player quando a aula possui quiz vinculado
- Bloqueio visual de próxima aula: indicador de "quiz obrigatório" enquanto não aprovado
- Questão dissertativa (`open`): aceita texto livre, não tem pontuação automática — marcada como "pendente de correção"

**OUT (o que esta story NÃO entrega):**

- Upload de tarefas / AssignmentUpload (→ Story 3.3)
- Correção manual de questões dissertativas pelo admin (→ Sprint 6 admin)
- Notificação por email ao aluno sobre resultado (→ Sprint 6 emails)
- Seed de quizzes com dados reais de aulas (→ separado, não é código)
- Sistema de relatórios de taxa de aprovação (→ Sprint 6)

## Acceptance Criteria

1. **AC1 — Quiz renderiza para aula com quiz:** Dado que a aula selecionada possui um quiz vinculado (`quiz.lesson_id = lesson.id`), quando o aluno visualiza a página do curso, o componente `<Quiz>` é exibido abaixo do player com o título do quiz e todas as questões.

2. **AC2 — Tipos de questão renderizados corretamente:** O componente `<Quiz>` renderiza:
   - `multiple_choice` — radio buttons com as opções do array `options`
   - `true_false` — radio buttons "Verdadeiro" / "Falso"
   - `open` — textarea para texto livre

3. **AC3 — Submissão e cálculo de nota:** Dado que o aluno responde todas as questões e clica em "Enviar", quando `POST /api/quiz/submit` é chamado com `{ quizId, answers }`, a API calcula a nota automaticamente para questões `multiple_choice` e `true_false` (ignora `open`), salva em `quiz_attempts` e retorna `{ score: number, passed: boolean, correctAnswers? }`.

4. **AC4 — Aprovação com base em pass_score:** A API marca `passed: true` quando `score >= quiz.pass_score` (padrão: 70). Retorna `passed: false` quando abaixo.

5. **AC5 — Componente QuizResult:** Após submissão, o componente `<QuizResult>` exibe: nota obtida, status (Aprovado ✓ / Reprovado ✗), feedback por questão quando `correctAnswers` está disponível, e botão "Tentar novamente" quando `attempts_remaining > 0`.

6. **AC6 — Limite de tentativas:** Dado que o aluno já usou todas as tentativas permitidas (`quiz.attempts_allowed`), quando tenta submeter novamente, a API retorna `400` com `{ error: 'Limite de tentativas atingido' }`. O componente oculta o botão "Tentar novamente".

7. **AC7 — Persistência da melhor tentativa:** Dado que o aluno já realizou o quiz, quando retorna à página do curso, o `<QuizResult>` exibe o resultado da última tentativa (não o formulário em branco), e o botão "Tentar novamente" aparece se ainda há tentativas disponíveis.

8. **AC8 — Bloqueio visual de próxima aula:** Na lista de aulas do `CoursePlayerLayout`, aulas que possuem quiz obrigatório e ainda não foram aprovadas exibem um ícone/indicador visual de "quiz pendente" diferente do estado normal.

9. **AC9 — Segurança da API:** A rota `POST /api/quiz/submit` verifica autenticação via Supabase session server-side. Usuário não autenticado recebe `401`. RLS impede que um usuário veja tentativas de outro usuário.

10. **AC10 — Validação Zod:** A rota `POST /api/quiz/submit` valida o body com Zod: `{ quizId: z.string().uuid(), answers: z.record(z.string(), z.string()) }`. Body inválido retorna `400`.

11. **AC11 — Typecheck e lint passam:** `npm run typecheck` e `npm run lint` executam sem erros após a implementação.

### RLS Relevante

CREATE POLICY "quizzes_read" ON quizzes FOR SELECT USING (true);
CREATE POLICY "questions_read" ON questions FOR SELECT USING (true);
-- quiz_attempts: apenas próprio record
CREATE POLICY "attempts_own" ON quiz_attempts FOR ALL USING (user_id = auth.uid());
```

### API Endpoint

POST /api/quiz/submit
  Body: { quizId: string; answers: Record<string, string> }
  Auth: Requer sessão + acesso à aula
  Response: { score: number; passed: boolean; correctAnswers?: Record<string,string> }
  Notas: Verifica attempts_allowed antes de aceitar nova tentativa
```

### Cálculo de Nota

- Apenas questões `multiple_choice` e `true_false` entram no cálculo
- `score = Math.round((acertos / total_gradable_questions) * 100)`
- Questões `open` recebem entry no answers mas não alteram o score
- Se `total_gradable_questions === 0` (quiz só com open): `score = 0`, `passed = false` (aguarda correção manual)
